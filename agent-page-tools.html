Agent Tool Behavior by Page Type

What an AI agent sees when it calls read_page or replace_lines / edit_sheet_cells on each PageSpace page type

text Returns plain/serialized text content with line numbers
structured Returns a rich JSON object (not raw text)
conditional Behavior varies based on state
blocked Operation not allowed; returns error
special Uses a different dedicated tool
📄 DOCUMENT HTML or Markdown Rich text document with formatting
📖 read_page
text Returns line-numbered content
HTML documents are run through addLineBreaksForAI — newlines injected after/before block tags (p, div, h1–h6, ul, li, etc.) so the AI can edit by line number. The function is additive only — no characters are removed, preserving mid-thought whitespace.
// HTML stored as:
"<p>Hello World</p><p>Second para</p>"

// Serialized for AI as (line-numbered):
1→<p>
2→Hello World
3→</p>
4→<p>
5→Second para
6→</p>
Markdown mode: if contentMode === 'markdown', content is passed raw (no HTML transformation). Line structure is natural.
Returned fields
content: "1→<p>\n2→Hello..." (line-numbered)
rawContent: "<p>\nHello..." (no numbers)
lineCount: 6
totalLines: 42
contentMode: "html" | "markdown"
isTaskLinked: false
stats: { wordCount, characterCount }
✏️ replace_lines
supported Full line-based editing
Both oldContent (diff baseline) and newContent (result) are normalized through addLineBreaksForAI identically, so a small edit diffs as a small change — not a full-document replacement. Replacement is spliced in at the specified line range. Empty replacement = deletion.
Returned fields
success: true
oldContent: "..." (normalized baseline)
newContent: "..." (after edit)
linesReplaced: 3
newLineCount: 44
changeType: "replacement" | "deletion"
Broadcasts a content-updated WebSocket event to all connected clients after every successful write.
💻 CODE raw text Code editor with syntax highlighting
📖 read_page
text Raw code, line-numbered
Code pages are treated as raw text — addLineBreaksForAI is not applied. This matters because code may contain raw HTML/XML (e.g., template literals, JSX) that the HTML injector would mangle. Each source line becomes one numbered line directly.
// Stored content:
"function greet() {\n return <div>Hi</div>;\n}"

// Returned as-is (line-numbered):
1→function greet() {
2→ return <div>Hi</div>;
3→}
✏️ replace_lines
supported Raw line editing (no HTML normalization)
Same line-splice logic as DOCUMENT, but isRawText = true so the content is not passed through addLineBreaksForAI. What the agent writes is stored verbatim, preserving indentation and avoiding HTML injection into source code.
Returned fields
success: true
type: "CODE"
changeType: "replacement" | "deletion"
🎨 CANVAS raw HTML/CSS Custom HTML/CSS standalone page
📖 read_page
text HTML serialized with line breaks
Canvas stores raw HTML/CSS as its content. Because contentMode is 'html' (not 'markdown') and Canvas is not a code page, it goes through addLineBreaksForAI. The AI sees the full HTML source with injected newlines around block tags.
✏️ replace_lines
supported Line editing with HTML normalization
Treated identically to DOCUMENT: HTML normalization applied to both old and new content. The agent writes raw HTML back — whatever it replaces is persisted as-is.
Canvas content is custom HTML. Agents writing to it should be careful not to break the page's layout structure (scripts, style blocks, etc.).
💬 CHANNEL structured transcript Team discussion channel
📖 read_page
structured Returns message transcript
Does not return the page's content column. Instead fetches all active channelMessages ordered by time, extracts text from the JSON message parts (prefers originalContent → parts[].text → textParts), and returns a numbered transcript.
// Each message line formatted as:
N→[user|agent|assistant] Name (ISO timestamp): text

// Returned fields:
channelMessages: [{ id, lineNumber, createdAt, senderName, senderType, content }]
messageCount: 5
totalMessages: 42
content: "1→[user] Alice (...): hello\n..."
lineStart / lineEnd map to message numbers, not character lines. Range requests slice the message list.
✏️ replace_lines
blocked Cannot edit channel messages via replace_lines
Channels don't have editable line content — messages are append-only records. The replace_lines tool has no channel-specific block, but since channels have no meaningful text content column the line operations would act on empty content.
How to write to a channel
Use the dedicated send_channel_message tool. This posts a new message record with proper attribution and real-time broadcast.
🤖 AI_CHAT structured (system prompt) AI agent conversation space
📖 read_page
text Returns the agent's system prompt / config as text
AI_CHAT pages store their config (system prompt, model, enabled tools) as content. read_page falls through to the generic text path and serializes whatever is in the content column — typically JSON-encoded config. It does not return chat history.
To read conversations, use list_conversations + read_conversation tools instead.
Conversation reads
list_conversations returns metadata (message count, timestamps, participants, first-message preview) for all conversations on this agent page.

read_conversation returns the actual messages formatted as N→[assistant|user|user@AgentName] text with 500-char truncation per message.
✏️ replace_lines
conditional Edits the agent config, not conversations
replace_lines would write to the content column (the system prompt / JSON config). This is technically possible but non-idiomatic — conversations themselves cannot be edited this way.
Idiomatic config editing
Use update_agent_config to set the system prompt, model, provider, and enabled tools for an AI_CHAT page. This is the correct path; it validates the config shape and writes structured fields.
✅ TASK_LIST structured task data Table-based task management
📖 read_page
structured Returns full task board, not raw content
Entirely bypasses the text-serialization path. Fetches tasks from the taskItems + pages join, resolves status configs (or falls back to defaults), computes sub-task counts, and returns a structured object.
type: "TASK_LIST"
taskListId: "abc123"
tasks: [{
  id, title, status, priority,
  position, assigneeId, dueDate,
  completedAt, linkedPageId,
  subTaskCount, subTaskCompletedCount
}]
availableStatuses: [{ slug, label, group, color }]
progress: { total, percentage, byGroup, bySlug }
parentTaskList: // if nested
If no task_lists record exists, one is auto-created on first read.
✏️ replace_lines
blocked Not meaningful for task lists
Task list pages have no editable text content column to write to. The task data lives in the taskItems table, not in the page content field.
Task management tools
Use dedicated task tools:
create_task — add a new task item
update_task — change status, priority, assignee, due date
delete_task — remove a task
reorder_task — change position
Each task has a linked page (its linkedPageId). Do not trash_page a task-linked page directly — it will break the task record. Use delete_task instead.
📊 SHEET TOML cell data Spreadsheet with formulas
📖 read_page
text Returns raw TOML cell data, line-numbered
Sheet content is stored as TOML (not HTML). read_page falls through to the generic serializer path — it applies addLineBreaksForAI (because contentMode is 'html' and SHEET is not a code page). The AI sees TOML source with injected newlines, which is not ideal for semantic understanding.
// The agent sees raw TOML like:
1→[sheet]
2→version = 1
3→rows = 10
4→cols = 5
5→
6→[[cells]]
7→address = "A1"
8→value = "Name"
For semantic sheet understanding, read_page then parse addresses manually, or use edit_sheet_cells which does the parse/serialize round-trip internally.
✏️ replace_lines vs edit_sheet_cells
blocked replace_lines is rejected for sheets
replace_lines detects isSheetType(page.type) and returns an error before touching the content. The error message explicitly redirects the agent.
success: false
error: "Cannot use line editing on sheets"
suggestion: "Use the edit_sheet_cells tool..."
special edit_sheet_cells — the correct write path
Takes an array of { address: "A1", value: "..." } updates. Internally: parse TOML → apply cell updates → serialize back to TOML → write to DB. Values starting with = are treated as formulas. Empty string clears a cell.
// Input:
cells: [{ address: "B2", value: "=A1*2" }]

// Returns:
cellsUpdated: 1
stats: { formulasSet: 1, valuesSet: 0, cellsCleared: 0 }
updatedCells: [{ address: "B2", type: "formula" }]
📎 FILE conditional (processing state) Uploaded file with extracted content
📖 read_page
conditional Depends on processingStatus
FILE pages have a processingStatus field that determines what the agent gets back:
pending / processing
success: false
error: "File is still being processed"
suggestion: "Please try again in a moment"
visual (image/PDF with no extractable text)
If the model supports vision: returns metadata (mimeType, fileSize, requiresVision flag) — the actual image bytes are passed in the context elsewhere.
If the model lacks vision: returns a helpful message listing suggested vision-capable models.
type: "visual_requires_vision_model"
suggestedModels: ["claude-3-5-sonnet", ...]
failed
success: false
error: "Failed to extract content"
processingError: "..."
completed — falls through to text serializer
Extracted text is in the content column. Serialized like a DOCUMENT (HTML normalization applied). Returns file metadata alongside text content.
fileMetadata: {
  mimeType, fileSize,
  originalFileName,
  processingStatus: "completed",
  extractionMethod, extractionMetadata
}
✏️ replace_lines
blocked FILE pages are always read-only
replace_lines checks page.type === 'FILE' before any permission check and returns an error immediately.
success: false
error: "Cannot edit FILE pages"
message: "File content is read-only and managed by the system."
suggestion: "Create a new document page instead."
📁 FOLDER minimal content Hierarchical organization container
📖 read_page
text Falls through to text serializer
Folders have no special read path — they fall through to the generic serializer. Their stored content is JSON ({"children":[]}), which after HTML normalization is returned as a single line. There is essentially no useful text content for the AI to read in a folder page.
// Typical result:
1→{"children":[]}

lineCount: 1
totalLines: 1
To understand a folder's contents, use list_pages on the drive, not read_page on the folder. Folder pages are purely structural.
✏️ replace_lines
not meaningful Technically allowed but writes to JSON
replace_lines has no folder-specific block. If an agent calls it on a folder, it would attempt to write to the {"children":[]} JSON — corrupting the folder's internal structure. There is no good reason to do this.
🖥️ TERMINAL experimental Interactive terminal (experimental)
📖 read_page
text Returns JSON history blob
Terminal pages store a JSON history ({"history":[]}) as content. No special read path — falls through to the generic serializer. After HTML normalization the agent sees the raw JSON, which includes terminal session history if any commands have been run.
✏️ replace_lines
not meaningful Would corrupt history JSON
No terminal-specific block in replace_lines. Writing to a terminal page would overwrite the history JSON. The terminal is driven by real-time WebSocket commands, not content edits.
